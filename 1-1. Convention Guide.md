# 코딩 컨벤션 가이드 (EDAS Gathering Server)

---

## 1. 목적과 적용 범위

- **목적**: 가독성, 유지보수, 로깅·예외 처리 일관성, 테스트 가능성을 맞추기 위한 규칙을 명시합니다.
- **적용**: `RawData` 솔루션 내 C# 코드 및 단위 테스트(`Tests`)에 적용합니다.
- **언어**: 런타임·API 문자열은 기존 시스템과의 호환을 위해 영문을 유지하고, **의도 설명 주석은 한국어**를 사용할 수 있습니다.

---

## 2. 솔루션·폴더 구조

| 영역 | 폴더(예) | 역할 |
|------|-----------|------|
| 진입·조율 | `RawData/main/` | 획득 시작, 스레드/태스크 로딩 |
| 장비 태스크 | `RawData/equip_task/` | 장비별 실행 루프(예: `CanonTask`) |
| 로그 태스크(베이스) | `RawData/log_CanonTask/` | 절차 클래스가 상속하는 베이스 |
| 로그 절차(구현) | `RawData/log_CanonTaskProcedure/` | FTP·파일 등 구체 구현, `partial` 분리 |
| 유틸 | `RawData/util/` | 폴더·파일 처리 등 공통 유틸 |
| 테스트 | `Tests/` | MSTest 기반 단위 테스트 |

**권장**: 기능 단위로 폴더를 나누고, **베이스(도메인 루프)** 와 **절차(인프라/IO)** 를 파일·네임스페이스로 분리합니다.

---

## 3. 네임스페이스

- 루트: `RawData`
- 하위: 폴더 역할과 대응 (예: `RawData.main`, `RawData.equip_task`, `RawData.util`, `RawData.log_Canontask`, `RawData.Log_CanontaskProcedure`)
- 테스트: `Tests`

**규칙**: 새 코드 추가 시 **폴더 경로와 네임스페이스를 일치**시키고, `using`은 아래 순서를 따릅니다.

1. `System.*` 계열
2. *(빈 줄)*
3. 프로젝트 내부(`RawData.*`)
4. *(빈 줄)*
5. 외부/레거시 어셈블리(`Lib_*` 등)

---

## 4. 접근 제한자와 타입 설계

- **구현 클래스**: 기본적으로 `internal` (예: `AcquisitionManager`, `CanonTask`, 절차 클래스).
- **상속**: 로그 루프·가상 훅은 베이스에서 `public virtual` / `protected`로 두고, 절차 클래스에서 `override`합니다.
- **XML 문서 주석**: 공개 의도가 중요한 타입·테스트 전용 API에는 `/// <summary>`를 둡니다 (예: 베이스 클래스 설명, `TestHooks` partial의 테스트 훅).

---

## 5. 명명 규칙

| 대상 | 규칙 | 예 |
|------|------|-----|
| 클래스, 구조체, 메서드, 프로퍼티, 이벤트 | PascalCase | `AcquisitionManager`, `PropertiesLoad`, `FileNameWrite` |
| 필드(인스턴스·정적 비상수) | camelCase | `usedTaskList`, `canonAdcCompDownloadPath` |
| 상수 | 의미 있는 PascalCase | `DefaultScanTime`, `TaskNameCanon` |
| 지역 변수·매개변수 | camelCase | `taskName`, `filePath` |
| `log4net` 로거 필드 | `log` (소문자), `static readonly` | `private static readonly log4net.ILog log = ...` |

**테스트**: 테스트 클래스·메서드는 의도가 드러나도록 긴 이름을 허용합니다 (예: `ADCCOMP_UploadPath_Build_Should_Match_Expected_Format`).

**레거시 혼재**: 일부 메서드가 PascalCase가 아닌 camelCase로 남아 있을 수 있습니다. **새 코드는 PascalCase**로 맞추고, 수정 시 해당 영역을 점진적으로 통일하는 것을 권장합니다.

---

## 6. 포맷·중괄호·공백

- **중괄호**: K&R이 아닌 **Allman 스타일** — 타입·메서드의 여는 중괄호 `{`는 새 줄에 둡니다.
- **한 칸 들여쓰기**: 프로젝트 기존과 동일하게 **스페이스 4칸**을 사용합니다.
- **`if` / `for` / `while`**: 본문이 한 줄이어도 **중괄호로 블록 처리**하는 패턴을 권장합니다 (가독성·디버깅·추후 변경 안전).

---

## 7. 로깅 (log4net)

각 클래스에서 공통적으로 다음 패턴을 사용합니다.

- 로거: `LogManager.GetLogger(typeof(현재타입))`
- 정보: `private void LogInfo(string message)` — 메시지 앞에 `[클래스명]` 접두사
- 오류: `private void LogError(string operation, Exception ex, object context = null)` — `ErrorFormat`으로 연산 이름·컨텍스트·예외 문자열 정리

**메시지 규칙**: `[클래스명]` + 공백 + 사용자에게 의미 있는 설명. 운영 로그 검색 시 클래스명으로 필터링하기 쉽게 합니다.

---

## 8. 예외 처리

- **초기화·스레드 시작 등 “한 단계 실패해도 나머지 진행”** 이 필요한 경우: `catch`에서 로깅 후 **삼키되**, **한국어 주석으로 의도를 반드시 남깁니다**.  
  예: *“초기화 단계 예외는 로깅 후 삼키기 (다른 스레드 초기화 계속 진행)”*
- **호출자가 실패를 판단해야 하는 경우**: `false` 반환 또는 로깅 후 **`throw`로 재전파** — 재던지기 전에도 로그를 남깁니다.
- **파일/IO**: 가능하면 구체 예외(`IOException` 등)를 구분해 로깅합니다.

---

## 9. `partial` 클래스와 테스트 훅

- 대용량 절차 클래스는 **`*.TestHooks.cs`** 같은 partial 파일로 분리합니다.
- 테스트 전용 팩토리 주입(예: `WebClient` 생성), 순수 함수 형태의 경로 조합 API 등은 **`internal`** 로 두고 XML 주석으로 “테스트 전용”임을 명시합니다.
- 프로덕션 코드 경로에서는 팩토리/헬퍼를 통해 테스트에서 대체 가능하도록 합니다.

---

## 10. 단위 테스트 (MSTest)

- 어셈블리: `Microsoft.VisualStudio.TestTools.UnitTesting`
- 클래스에 `[TestClass]`, 메서드에 `[TestMethod]`
- **Arrange / Act / Assert** 주석으로 단계를 구분합니다.
- 검증은 `Assert.AreEqual` 등으로 기대값을 명확히 둡니다.

---

## 11. 상수·매직 값

- 반복되는 문자열·간격(ms) 등은 **`private const`** 또는 의미 있는 이름의 상수로 뺍니다 (예: 다운로드 폴링 간격 `DownloadCheckIntervalMilliseconds`).
- 리플렉션용 타입 이름 문자열 등은 한 곳에 모아 **오타·변경 시 한 번에 수정** 가능하게 합니다.

---

## 12. `#region`

- 긴 클래스에서는 **관련 메서드 묶음**에 `#region`을 사용해 탐색을 돕습니다.  
  과도한 중첩 region은 피하고, “절차 단계별(다운로드/업로드/헬퍼)” 수준을 권장합니다.

---

## 13. 파일 이름과 타입 이름

- **이상적 규칙**: 파일명 = 주 타입명 (`Canon_ADCCOMPProcedure.cs` ↔ `Canon_ADCCOMPProcedure`).
- **정리 권장**: 파일명과 클래스명 대소문자가 다른 경우(예: manager 표기)는 리팩터링 시 **파일명 변경**으로 맞춥니다.
