# 🚀 진행 시 문제점
**개발자들의 개발 직렬화**

## 1. 문제 원인 분석
초기 역할 분담에도 불구하고 개발 효율이 저하되었던 핵심 원인들입니다.

| 구분 | 주요 원인 | 상세 내용 |
| :--- | :--- | :--- |
| **설계 결함** | **강한 결합도** | 인터페이스 기반 설계 부재로 인해 객체 간 의존성이 높음 |
| **진행 병목** | **개발의 직렬화** | 상대방의 코드가 완성되어야 내 코드를 짤 수 있는 대기 상태 반복 |
| **추상화 부족** | **내부 구현 의존** | 기능을 호출할 때 상대방의 내부 로직까지 알아야 하는 상태 |
| **테스트 한계** | **통합 테스트 편중** | 개별 검증이 불가능하여 결국 합쳐서 테스트하는 1인에게 업무 몰림 |

---

## 2. 향후 보강 전략

### 🎯 핵심 방향
> **"상대방의 코드 완성 여부와 상관없이, 내 담당 구역을 독립적으로 개발하고 테스트할 수 있는 환경 구축"**

### 🛠 실행 방안

#### **① Interface 중심 설계 (Contract-First)**
* 구현체(`Class`)를 만들기 전에 **인터페이스(`Interface`)**를 먼저 정의합니다.
* 인터페이스는 일종의 **'규약(Contract)'**입니다. 이 규약만 합의되면 내부 로직이 비어 있어도 협업이 가능합니다.

#### **② 모킹(Mocking) 활용**
* 다른 개발자의 기능이 미완성 상태라면, 미리 정의된 인터페이스에 **가짜 객체(Mock)**를 끼워 넣습니다.
* 이를 통해 외부 의존성 없이 자신의 로직을 독자적으로 실행할 수 있습니다.

#### **③ 단위 테스트(Unit Test) 강화**
* **통합 테스트 전 단계:** 전체를 합쳐보기 전에 본인이 담당한 모듈이 인터페이스 규격에 맞게 작동하는지 개별 검증합니다.
* **격리된 테스트:** 테스트 코드가 남의 코드에 의존하지 않도록 격리하여 작성합니다.

## 3. 개선된 워크플로우 (Proposed Workflow)

1.  **정의 단계**: 기능을 나누고 각 기능이 주고받을 **Input/Output(Interface)**을 먼저 확정합니다.
2.  **병렬 단계**:
    * **A**: Mock 객체를 활용해 기능 구현 및 단위 테스트 진행
    * **B**: Mock 객체를 활용해 기능 구현 및 단위 테스트 진행
3.  **통합 단계**: 실제 구현체들을 연결하여 전체 시나리오를 검증(Integration Test)합니다.
