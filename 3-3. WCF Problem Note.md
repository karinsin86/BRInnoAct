# 🚀 진행 시 문제점
**개발자들의 개발 직렬화**

## 1. 문제 원인 분석
초기 계획했던 '역할별 병렬 완성'이 실무에서 '순차적 직렬 구현'으로 변질된 핵심 원인들입니다.

| 구분 | 주요 원인 | 상세 내용 |
| :--- | :--- | :--- |
| **구조적 한계** | **단일 통합 지점** | 서비스 계약(WCF, DTO 등)이 하나로 묶여 있어 뼈대를 잡는 사람에게 작업이 쏠림 |
| **공정 간섭** | **환경 의존성 병목** | DB 연결, 호스팅 환경 등 공통 인프라가 준비되기 전까지 타 인원 작업 불가 |
| **심리적 요인** | **통합 불확실성** | 개별 구현보다 '동작하는 한 줄기'를 같이 확인하며 가는 것이 안전하다고 느낌 |
| **운영 이슈** | **머지(Merge) 부담** | 동시 수정 시 발생하는 충돌을 피하기 위해 무의식적으로 '한 명씩' 작업 수렴 |
| **정의 부재** | **완료 기준(DoD) 모호** | '내 PC에서 컴파일됨'과 '전체 시스템에서 동작함' 사이의 간극을 1인이 메움 |

## 2. 향후 보강 전략 (Recovery Strategy)

### 🎯 핵심 방향
> **"결합도를 낮추어 병렬성을 확보하고, 통합(Integration)의 책임을 시스템과 프로세스로 분산"**

### 🛠 실행 방안

#### **① 병렬 가능한 최소 단위 설계 (Vertical Slicing)**
* **인터페이스 격리**: WCF 같은 외부 노출 계층과 내부 로직(DB, 파일 핸들러) 사이에 순수 C# 인터페이스 경계를 둡니다.
* **가짜 구현 활용**: 인터페이스만 확정되면 실제 로직이 없어도 Mock 객체를 통해 각자 파트를 동시에 개발합니다.

#### **② 완료 정의(DoD)의 격상**
* **통합 중심 완료**: 기능 구현의 끝을 '내 브랜치 작업 완료'가 아닌, '공통 브랜치 머지 및 최소 1개 시나리오 통과'로 정의합니다.
* **CI(지속적 통합) 도입**: 빌드와 테스트를 자동화하여 "나중에 몰아서 테스트"하는 관행을 방지합니다.

#### **③ 환경 선행 및 문서화**
* **인프라 우선 순위**: DB 설정, UNC 경로 등 공통 환경 구축을 스프린트 최우선 과제로 배정하고 재현 스크립트를 공유합니다.

#### **④ 짧은 호흡의 머지(Short-lived Branches)**
* **작은 단위 커밋**: 기능을 1~2일 단위의 얇은 조각으로 나누어 자주 머지함으로써 대규모 충돌 위험을 줄이고 진행 상황을 가시화합니다.

## 3. 개선된 워크플로우 (Proposed Workflow)

1.  **계약 확정 (Design First)**: 회의를 통해 서비스 계약(Interface)과 데이터 구조(DTO)를 최우선으로 고정합니다.
2.  **병렬 구현 (Parallel Development)**:
    * **역할 A**: 인터페이스를 소비하는 클라이언트/호스트 로직 구현 (Mock 사용)
    * **역할 B/C**: 인터페이스를 구현하는 세부 비즈니스 로직 및 단위 테스트 진행
3.  **상시 통합 (Continuous Integration)**: 
    * 작은 단위로 메인 브랜치에 합치며 **머지 게이트(Merge Gate)**를 통해 실시간으로 정합성을 검증합니다.
4.  **회고 (Retrospective)**:
    * 작업이 직렬화되었다면 원인이 **환경**이었는지, **설계**였는지 파악하여 다음 차수에 반영합니다.
